# 임팩트랩 방문 기록 등록 시스템 — 배포 후속 작업 목록

v3(담당자/기관 중심 데이터 모델 + 캘린더 자동등록) 적용 이후 남은 작업을 정리한 문서.
관련 코드: `apps-script/Code.v3.gs`, `api/register-visitor.js`, `README.md`

## 1. Google Sheets 준비

- [ ] 방문기록 시트를 A~N 컬럼(등록일시 ~ 캘린더 동기화 상태)으로 재구성
      — 기존 방문자 개인정보 컬럼이 있다면 별도 탭으로 백업 후 정리
- [ ] 카페이용내역 탭 존재 확인 (없으면 생성: 등록코드 / 등록일시 / 이용메뉴 / 인원수)
- [ ] 카페메뉴설정 탭 생성 및 초기 메뉴 데이터 입력 (메뉴명 / 노출여부 / 정렬순서)
- [ ] 공유 캘린더 생성 (예: "임팩트랩 방문일정") → 담당자들이 볼 수 있도록 공유 설정

## 2. Apps Script 병합 및 배포

- [ ] 운영 중인 Apps Script 편집기에서 `Code.v3.gs` 내용을 기존 코드에 병합
      (`doPost`, `handleAddVisit`, `tryCreateCalendarEvent` 등)
- [ ] Script Properties에 `CALENDAR_ID` 신규 등록 (`SECRET_TOKEN`은 기존 값 유지)
- [ ] 프로젝트 시간대가 Asia/Seoul인지 확인
- [ ] 웹앱으로 새 버전 배포 → 배포 URL이 바뀌면 Vercel의 `GAS_WEBHOOK_URL`도 갱신
- [ ] 재배포 시 캘린더 접근 권한 재승인 (최초 1회)
- [ ] (신규) `Code.dashboard.gs`를 **별도의 새 Apps Script 프로젝트**에 배포
      — Apps Script A와 같은 프로젝트에 넣지 말 것 (`doPost` 충돌)
- [ ] (신규) 대시보드 프로젝트 Script Properties에 `DASHBOARD_SECRET_TOKEN`,
      `SPREADSHEET_ID` 등록

## 3. Vercel / GitHub 배포

- [ ] 변경된 코드 전체를 GitHub 리포지토리에 push
- [ ] Vercel 환경변수(`GAS_WEBHOOK_URL`, `GAS_SECRET_TOKEN`) 최신 값인지 확인
- [ ] (신규) Vercel 환경변수(`GAS_DASHBOARD_WEBHOOK_URL`, `GAS_DASHBOARD_SECRET_TOKEN`) 등록
- [ ] 재배포 후 `{프로젝트명}.vercel.app` 접속해서 폼이 정상 렌더링되는지 확인
- [ ] (신규) `{프로젝트명}.vercel.app/dashboard.html` 접속해서 대시보드가 정상 렌더링되는지 확인

## 4. 기능 테스트

- [ ] 담당자 이메일이 `@gbcommons.org`가 아니면 등록이 거부되는지
- [ ] 정상 등록 시 방문기록 시트 A~N에 값이 올바르게 들어가는지
- [ ] 카페 이용 체크 시 카페이용내역 시트에도 함께 기록되는지 (메뉴는 빈 값) / 체크 안 하면 방문기록에만 기록되는지
- [ ] 캘린더에 제목·설명·시간이 올바른 이벤트가 생성되는지
- [ ] 같은 브라우저로 재접속 시 담당자 이메일/이름이 자동으로 채워지는지 ("정보 수정" / "삭제" 버튼 포함)
- [ ] 짧은 시간에 반복 제출 시 rate limit(10초당 5회)이 걸리는지
- [ ] (신규) 대시보드에서 카페 이용 신청 목록이 담당자/방문기관명/인원수와 함께 표시되는지
- [ ] (신규) 메뉴 미입력 건이 목록 위쪽에 정렬되는지
- [ ] (신규) 대시보드에서 메뉴 입력 후 저장 시 카페이용내역 시트 C열이 갱신되고 카드가 "입력완료"로 바뀌는지
- [ ] (신규) 등록 폼에서 카페 이용 체크 시 더 이상 메뉴 선택 UI가 뜨지 않고 인원수만 입력받는지

## 5. 운영/접근 관리

- [ ] Vercel 배포 URL을 담당자들에게만 내부적으로 공유
      (검색엔진 노출 방지 차원에서 `robots.txt`나 URL 난수화도 고려 가능)
- [ ] 필요하면 이후 간단한 접근 코드/비밀번호 게이트 추가 검토
- [ ] (신규) 대시보드는 인증 없이 URL 공유만으로 접근하도록 결정됨 — 배포 시
      `dashboard.html` 대신 추측하기 어려운 경로명으로 바꾸는 것을 권장

## 6. 놓치기 쉬운 부분

- [ ] 기존에 이미 수집돼 있던 방문자 개인정보(이름/연락처 등)를 어떻게 처리할지 결정
      — 그대로 두면 개인정보보호법상 파기 시점 이슈가 있을 수 있어, 삭제 또는 별도 보관
      정책 검토 필요 (법률 자문이 필요할 수 있는 부분)
- [ ] 방문 목적별 캘린더 기본 소요시간(`DURATION_BY_PURPOSE_MIN`)이 실제 운영 패턴과 맞는지 검토
